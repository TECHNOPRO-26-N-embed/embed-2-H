## 1. システム全体像

### 1-1. ブロック構成（入力 → 処理 → 出力）

> ※ Arduinoを中心に「何が入力で何が出力か」をテキストの図で整理します。

**記入例：**
```
【入力】                 【Arduino UNO R3】            【出力】
タクトスイッチ ─────→┐                         ┌──→ LED（赤・緑）
超音波センサー ─────→┤   状態管理・判定処理    ├──→ パッシブブザー
                      └─────────────────────────┘
```

**↓ 自分のシステムを書いてください**
```
【入力】                 【Arduino UNO R3】            【出力】
タクトスイッチ ─────→┐                         ┌──→ LCD表示（WAIT/PUSH/結果）
               ├─ 状態管理・判定処理 ───┤
               └─────────────────────────┘
                               └──→ パッシブブザー
```

補足説明（図で表現しきれない場合）：

- 入力はタクトスイッチ1つとし、状態に応じてLCD表示とブザー音を切り替える。
- 反応時間は `micros()` で計測し、表示時にmsへ換算する。

### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON / 初期化]
  →（初期化完了）
[開始待ち] ──（ボタン押下）──→ [READY表示] ──（約0.8秒）──→ [待機状態]
[待機状態] ──（ランダム時間経過）──→ [合図状態] ──（ボタン押下）──→ [結果表示]
[結果状態] ──（一定時間経過 または ボタン押下）──→ [開始待ち]
└────────────────────（リセットボタン）─────────────────────┘

※フライング時
[待機状態] ──（ボタン押下）──→ [失敗状態] ──（一定時間後）──→ [開始待ち]
```


**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| 初期化 | 起動直後の準備処理 | LCD初期化のみ（表示は開始待ちへ） |
| 開始待ち | スタート入力待ち | LCD：「Press button to start」表示 |
| READY表示 | ゲーム開始直前の案内 | LCD：「READY」表示 |
| 待機状態 | ランダム時間待機 | LCD：「WAIT...」表示|
| 合図状態 | 押すタイミング提示 | LCD：「PUSH!」表示　Passive Buzzer：短音 |
| 結果状態 | 反応時間表示 | LCD：「Time:〇〇ms」 , 「Best:○○ms」|
| 失敗状態 | フライング判定 | LCD：「MISS」表示 Passive Buzzer：失敗した音|

> [!TIP]
> すべての状態から「抜け出せるルート」があるか確認してください。
> 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| ボタン監視 | 常時 | **millis** | 即応性が必要 |
| ランダム待機 | 2〜5秒 | **millis** | WAIT中もフライング判定を継続するため |
| 表示更新 | 必要時 | **millis** | 他処理と並行 |
| ブザー制御 | 短時間 | **millis** | ボタン監視を止めないため |

> [!CAUTION]
> `delay()` を使う場合、その間はボタン入力もセンサー読み取りも**すべて止まります**。

## 4. 異常系・エラー処理の方針

| 異常ケース | 発生条件 | 検知方法 | 対応方針 |
|:--|:--|:--|:--|
| チャタリング | ボタン連打 | 前回入力から50ms未満の入力を検出 | デバウンス処理でスキップ |
| フライング入力 | WAIT表示中にボタン押下 | `checkFalseStart()` が true を返す | MISS表示と失敗音を出して待機へ戻る |
| LCD未接続/表示失敗 | 起動時に表示更新できない | 初期表示が出ない | シリアルへエラーログを出し、最小動作を継続 |
| ベスト記録未初期化 | 初回計測時 | `bestTimeUs == 4294967295` | 初回値をそのままBestとして登録 |
| ボタン押下なし | 合図後に一定時間入力なし | タイムアウト経過を検出 | 「TIMEOUT」表示後に待機へ戻る |

## 7. AIレビュー記録

> グループレビューの前に必ず実施してください。

### Q1: 設計の妥当性確認

> 「この基本設計書をもとに Arduino UNO R3 でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay() の問題点はありますか？」

**AIの回答（要約）：**

- ピン競合はなし（ボタンD9、ブザーD8、LCDはD2-D7で重複なし）。
- ライブラリは `LiquidCrystal` を使用する前提で問題なし（Arduino標準ライブラリ）。
- 状態遷移は WAIT → GO → RESULT と、フライング時の MISS が定義されており、基本動作は成立。
- `delay()` を使うと入力監視が止まるため、待機・監視系は `millis()` ベースを維持するのが妥当。

**対応した内容：**

- ボタン入力をD9 + `INPUT_PULLUP` に統一し、配線と判定仕様（押下=LOW）を明記した。
- ランダム待機・ボタン監視・ブザー制御を `millis()` 方針で統一した。
- LCDをI2C前提からパラレル接続（D2-D7）へ変更し、設計書全体を更新した。
- トレーサビリティ表と結合テスト表を必須機能4件に対応させて整理した。

---

### Q2: チャタリング・delay・ノイズの確認

> 「この設計にチャタリング対策は含まれていますか？delay() を使っている箇所で問題はありますか？アナログセンサーを使う場合、ノイズ対策は必要ですか？」

**AIの回答（要約）：**

- チャタリング対策は、50msデバウンス（前回入力時刻との差分判定）として設計に含まれている。
- `delay()` 依存にするとフライング判定や入力応答が落ちるため、本件は非ブロッキング（`millis()`）が適切。
- 本システムはアナログセンサーを使っていないため、アナログノイズ対策は必須ではない。
- ただし電源・GND共通化、配線の引き回し、LCD16ピン配線ミス対策は実機で重要。

**対応した内容：**

- 異常系を実装対象に合わせて更新し、センサー前提の記述を削除した。
- フライング入力、LCD表示失敗、ベスト初期値、入力タイムアウトを異常系として追加した。
- ボタン配線を `INPUT_PULLUP` に合わせ、外付け抵抗不要の方針へ修正した。
- LCDをI2C接続からパラレル接続へ変更し、可変抵抗によるコントラスト調整を追記した。
